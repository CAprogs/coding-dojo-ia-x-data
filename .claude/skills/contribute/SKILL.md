# Contribution Rules

Follow these rules whenever contributing code to this project. They are enforced by CI and pre-commit hooks — violating them will break the build.

## 1. Conventional Commits (enforced by commitizen)

Every commit message MUST match this pattern:

```
<type>(<scope>)?<!>?: <subject>

<body?>

<footer?>
```

### Allowed types

| Type       | When to use                                                              |
| ---------- | ------------------------------------------------------------------------ |
| `feat`     | New feature (minor version bump)                                         |
| `fix`      | Bug fix (patch version bump)                                             |
| `docs`     | Documentation only (README, comments, guides)                            |
| `style`    | Formatting / whitespace — no logic change                                |
| `refactor` | Code restructure without behavior change                                 |
| `perf`     | Performance improvement                                                  |
| `test`     | Adding / fixing / removing tests                                         |
| `build`    | Build system or dependencies (uv, poetry, pyproject.toml)                |
| `ci`       | CI/CD config (GitHub Actions, workflows)                                 |
| `chore`    | Housekeeping (e.g. version bump by `cz bump`, cleanup)                   |
| `revert`   | Reverting a previous commit                                              |

### Rules

- **Subject**: imperative mood, present tense, lowercase start, no trailing period. Max ~72 chars.
  - Good: `feat: add handicap_friendly_events gold model`
  - Bad: `feat: Added handicap model.` / `feat: added something.`
- **Scope** (optional): a short noun in parens identifying the area, e.g. `feat(ingestion):`, `fix(dbt):`, `ci(lille):`.
- **Breaking changes**: append `!` after type/scope (`feat!: drop Python 3.11 support`) AND add a `BREAKING CHANGE:` footer explaining the migration.
- **Body** (optional): wrap at ~100 chars, explain the *why*, not the *what*.
- **Footer** (optional): `Closes #123`, `Refs #456`, or `BREAKING CHANGE: ...`.

### How to commit

```bash
# Guided interactive commit (recommended if unsure)
uv run cz commit

# Manual commit — commitizen will validate the message via the commit-msg hook
git commit -m "feat(silver): deduplicate events by id"
```

If `git commit` is rejected, run `uv run cz check --message "<your msg>"` to see the error.

## 2. Branch Naming

Branches must follow this pattern: `<type>/<short-kebab-description>`.

- Use the same type set as commits: `feat/`, `fix/`, `docs/`, `refactor/`, `chore/`, `ci/`, `test/`, `perf/`, `build/`, `style/`.
- Include an issue number when relevant: `feat/42-add-postal-code-filter`.
- The `commitizen-branch` pre-push hook validates that commits on the branch follow conventional commits.

Never push directly to `main` or `lille` — always go through a PR.

## 3. Before Pushing

Always run the full quality gate locally:

```bash
uv run just quality-all
```

This runs all pre-commit hooks (trailing whitespace, yaml/toml checks, commitizen, ruff, mypy, sqlfluff) on every file. Fix every failure before pushing — the `lille` CI runs the same checks and will block the PR.

## 4. Pull Requests

### Title
The PR title MUST also be a conventional commit (`feat: ...`, `fix(ingestion): ...`). This is what ends up on `main` when the PR is squash-merged, so it feeds the changelog generated by `cz bump`.

### Description
Use this template:

```markdown
## Summary
- <bullet 1: what changed and why>
- <bullet 2>

## Test plan
- [ ] <how you verified the change>
- [ ] <edge cases covered>
- [ ] CI green

Closes #<issue>
```

### Size & scope
- One PR = one concern. Split feature + refactor into two PRs.
- Target ≤ 400 lines changed. If it's bigger, explain why in the description.
- Do NOT mix generated file changes (e.g. `uv.lock` regen) with unrelated feature work — call them out in the description.

### Target branch
- Feature / fix work → target `lille` (the integration branch, runs the full CI).
- Only promote `lille` → `main` via a dedicated release PR.

### CI
A PR is only mergeable when:
1. `lille CI` (pre-commit via prek) is green.
2. All review comments are resolved.
3. The branch is up to date with the base branch (rebase or merge).

## 5. Code Review

### As the author
- Self-review the diff on GitHub **before** requesting review — catch typos, leftover prints, commented code.
- Respond to every comment. Either push a fix or explain why you disagree; don't silently ignore.
- Prefer `fixup` commits during review, then squash at merge time. Avoid `git push --force` on a branch others are reviewing unless coordinated.

### As a reviewer
- Check: correctness, tests, performance-critical paths, SQL correctness (dbt models), logging discipline (use `src/logger/log_handler.py`, never `print`/`rprint`).
- Use `nit:` prefix for cosmetic comments that shouldn't block merge.

## 6. Releases (maintainers)

Version bumps and the CHANGELOG are handled by commitizen:

```bash
uv run cz bump --changelog
git push --follow-tags
```

This reads the commit history since the last tag, picks the correct semver bump (major/minor/patch) based on commit types, updates `pyproject.toml` version, and appends to `CHANGELOG.md`. Do not edit version numbers by hand.

## 7. Quick Reference

```bash
uv run cz info                       # Show the commit schema
uv run cz commit                     # Guided commit
uv run cz check --message "..."      # Validate a message
uv run cz bump --changelog           # Release: bump version + changelog
uv run just quality-all              # Run all pre-commit hooks
```
